# Neighbourhood Echoes — Project Handoff & Learnings

A social-mystery / life-sim game where NPCs **remember** what the player says and does,
powered by a [Cognee](https://docs.cognee.ai) memory service behind a game client.
This file is the master handoff: **status → how to run → learnings → next steps**.

> Design source: `Neighbourhood Echoes Development Plan.pdf`
> Phase-1 plan: `PHASE1_BACKEND_PLAN.md`
> Component docs: `backend/README.md`, `game-client/README.md`

## 1. Current status (2026-06-28)

Pivoted to the **React hackathon build** (see `Neighbourhood Echoes Hackathon Plan`):
a browser game with a clickable map, choice dialogue, case notebook, relationship
meters, and a Memory Debugger — replacing the earlier Godot prototype.

| Phase | What | State |
|---|---|---|
| **1** | Memory engine proves divergent NPC dialogue (Python CLI) | ✅ **Done & verified** |
| **1.5** | FastAPI layer over the memory engine | ✅ **Done & verified** |
| **2** | React + Vite + TS client (map, dialogue, notebook, debugger) | ✅ **Built & verified** |
| **2-write** | NPCs remember new things (promise/secret/gossip write-back) | ✅ **Built & verified** |
| **3** | Promises, two-day mystery, gossip spread, relationships, endings | ✅ **Built & verified** |
| validator | Locked-fact guardrail + authored fallback for dialogue | ✅ **Built** |

**Proven thesis:** each NPC recalls only its own Cognee dataset, so the same question
gets different answers (Maya protects Sam, Sam denies, Jules spreads a rumour). The
game now builds on that: promise Maya a secret, betray her to Jules, and on Day 2 the
rumour has **travelled through the neighbourhood** — Maya confronts you because the
betrayal memory reached her dataset. The flow was verified end-to-end over HTTP and
with a 38-check stubbed integration test (both endings, gating, gossip spread,
LLM path, and the guardrail).

## 3. How to run

### Backend (do this first — everything depends on it)
```bash
cd backend
.venv/bin/python ask.py --reset --seed            # seed memories (~10 LLM calls)
.venv/bin/python ask.py "Who broke into the shed?" --all          # generated answers
.venv/bin/python ask.py "Who broke into the shed?" --npc maya --context   # raw memories (debug)
```

### API (the game server)
```bash
cd backend
.venv/bin/uvicorn api:app --port 8000             # add --reload while developing
curl -s -X POST localhost:8000/npc/talk -H 'Content-Type: application/json' \
  -d '{"npcId":"maya"}'
```
Dialogue is generated **through Cognee recall**, so it runs in either mode (see
`backend/README.md`): **Cognee Cloud** (`COGNEE_SERVICE_URL`+`COGNEE_API_KEY`, billed to
credits, no LLM key) or **local** (Gemini key). No key at all → authored fallback lines.
For fast UI testing with no key/install: `.venv/bin/python devserver_stub.py`.

### Game client (React)
```bash
cd game-client
npm install
npm run dev                                        # http://localhost:5173
```
Open the URL, click **Start the Mystery**, and play. (Backend must be running on :8000;
without a Gemini key/seed the game still works on authored fallback lines.)

## 4. Learnings (the hard-won stuff — read before resuming)

### 4.1 Cognee (verified against v1.2.2 live)
- **Install:** `cognee==1.2.2`, Python **3.10–3.14**. Use **Python 3.12** — 3.14 lacks
  prebuilt wheels for some native deps (kuzu/lancedb/pyarrow).
- **Local defaults need no servers:** SQLite (relational) + LanceDB (vector) + Kuzu (graph).
- **High-level API:** `remember / recall / improve / forget`. `remember()` is self-contained
  (internally runs add → cognify → improve); `recall()` auto-routes to `GRAPH_COMPLETION`.
- **THE core mechanism — dataset isolation:** write with
  `remember(text, dataset_name="npc_maya_memory")`; read with
  `recall(query, datasets=["npc_maya_memory", ...shared])`. Scoping is **leakage-free** —
  this is exactly why NPCs diverge. Always pass `datasets=[...]` or it searches everything.
- **`only_context=True`** returns retrieved context only (no LLM answer, cheap). The default
  (`False`) returns a synthesized answer but generates **~one answer per retrieved source**
  (~3 per NPC) — wasteful. We switched dialogue to `only_context=True` + our own single LLM call.
- **`self_improvement=False`** when seeding to skip the enrichment pass (cheaper/faster).
- **Config:** env vars loaded *before* importing cognee — `LLM_PROVIDER/LLM_MODEL/LLM_API_KEY`,
  `EMBEDDING_PROVIDER/EMBEDDING_MODEL/EMBEDDING_DIMENSIONS/EMBEDDING_API_KEY`. Pin data dirs with
  `cognee.config.system_root_directory(...)` / `data_root_directory(...)`.
- **Reset:** `prune_data()` + `prune_system(metadata=True)` for a clean slate (no LLM cost),
  or just delete `.cognee_system/` + `.cognee_data/`.
- **Kuzu is single-writer** with a known file-lock bug — **serialize all cognee calls**
  (the API holds `memory.LOCK`). Don't run two cognee processes against the same dirs.

### 4.2 Gemini free tier (this bit us repeatedly)
- **Hard cap: ~20 generate-content requests PER DAY, PER MODEL** on the free tier
  (`GenerateRequestsPerDayPerProjectPerModel-FreeTier`). Not per-minute — per day.
- **`gemini-2.0-flash` is NOT free-tier enabled on this key (limit 0).** Working models:
  `gemini-2.5-flash`, `gemini-2.5-flash-lite`, `gemini-flash-latest`, `gemini-3.1-flash-lite`.
- **Embeddings:** use `gemini/gemini-embedding-001`, **default 3072 dims** — set
  `EMBEDDING_DIMENSIONS=3072` to match or the vector store breaks. (`text-embedding-004` is
  retired → 404.)
- **Mitigations applied:** batch each dataset's memories into ONE `remember()` call (cut the
  seed from ~70 LLM calls to ~10); rotate `LLM_MODEL` in `.env` when one model is exhausted.
- **The real fix: enable billing** on the Gemini project. This whole demo costs well under a
  cent on pay-as-you-go and removes the 20/day wall. Quotas otherwise reset at Pacific midnight.

### 4.3 Architecture / design
- **Backend-first was correct.** The risk (clean, scoped, fast recall) lives in the backend;
  the game client is a thin HTTP caller.
- **Separate retrieval from generation** (`only_context=True` + own LLM call). It's cheaper and
  leaves a clean seam to insert the design doc's response validator (PDF §24).
- **Confabulation is real and observed.** With the raw recall→LLM path, a lower-tier model
  (gemini-3.1-flash-lite) had Jules invent facts NOT in her datasets ("Maya lost the key",
  "Omar accusing Sam"). This confirms the PDF's **Risk 1** — you **need** the authored-fallback
  + canonical-fact validator before shipping. Phase 1/2 deliberately omit it.
- **Godot:** build the UI in code (autoload) and use built-in input actions — minimizes the
  manual editor wiring, since the human drives the editor and the assistant drives the code.

### 4.4 Environment (macOS / this machine)
- Use **Python 3.12** at `/usr/local/bin/python3.12` (3.14 is the default but breaks native wheels).
- Use a **real cloud LLM key** (not Ollama/local) to avoid cognee macOS hang issues
  (#1743 / #2119).
- **SECURITY:** the Gemini key lives in `backend/.env` (gitignored). It was pasted into a chat,
  so **rotate it** before any real/shared use.

## 5. Next steps (resume here)

### Do these first
1. **Re-seed the backend.** The local cognee DBs were cleared when the project moved into
   `npc-mem/` (they aren't portable). From `npc-mem/backend`: `.venv/bin/python ask.py --reset --seed`.
   (Pick a Gemini model in `.env` that still has daily quota — see §4.2.)
2. **Run the Godot client** and confirm the scene loads and dialogue works end-to-end.
   It's untested (Godot isn't installed here). If it errors, the Godot Output panel text is
   what's needed to fix it.
3. **Decide on Gemini billing.** Without it you'll keep hitting the 20/day/model wall during
   iteration. Enabling it (pennies) is the unblock; otherwise plan to rotate models.

### Then, completing Phase 2 → following the design doc's phase order
3. **Conversational write-back** (finishes Phase 2's "NPC remembers what you said"):
   after `/dialogue/respond`, store the player's line into the NPC's dataset.
   - Cheap path: cognee **session memory** (`session_id` per save/day) — avoids the heavy graph
     pipeline per turn. Heavy path: `remember(...)` gated by an importance score.
   - This is quota-heavy on the free tier → wants billing or session memory first.
4. **Importance scoring** (PDF §8) — score events 0–1 before deciding what to persist; don't
   store every line.
5. **Response validator / canonical fact-checker** (PDF §24) — sits between recall and
   generation; blocks confabulation, locked-secret leaks, invented NPCs. Needed before this is
   demoable to others.
6. **Phase 3 — Promises:** detect promise-like player choices, track deadlines, check
   fulfilment, apply relationship deltas, change dialogue if broken.
7. **Phase 4 — Mystery/clues:** contradiction-detection recall recipe, evidence objects, case
   notebook UI, accusation flow.
8. **Phase 5 — Gossip:** nightly spread of memories between NPC datasets with distortion.
9. **Phase 6 — Relationship-driven branching:** trust/suspicion gates open different paths.

### Cross-cutting (build alongside)
- **Numeric relationship state** lives in a normal game DB, NOT in the LLM/Cognee (PDF §5, §13).
- **Debug tools** (PDF §20): expand `/memory/recall` into a proper memory viewer + conversation
  trace. Invaluable for debugging "bad answer = bad storage vs retrieval vs prompt?".

## 6. Open questions / risks to watch
- **Recall latency & cost per dialogue turn** at scale (recall + generation, possibly +write-back).
  No latency budget set yet — measure once billing is on.
- **Kuzu concurrency** under the nightly NPC-to-NPC simulation (Phase 5) — may force the move to
  Neo4j earlier than "production" (PDF notes this; Cognee docs confirm Neo4j for multi-agent).
- **Validator quality** — confabulation means generated facts can't be trusted raw; the validator
  is on the critical path, not optional polish.
